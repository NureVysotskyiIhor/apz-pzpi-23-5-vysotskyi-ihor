     3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Загальні відомості про патерни проєктування
     Загальні відомості про патерни проєктування
          Патерни проєктування GoF — це 23 класичних шаблони, описані у 1994 році в книзі «Design Patterns: Elements of Reusable Object-Oriented Software» авторами Gamma, Helm, Johnson та Vlissides. Вони поділяються на три категорії:
          • Породжувальні — Abstract Factory, Builder, Factory Method, Prototype;
          • Структурні — Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy;
          • Поведінкові — Chain of Responsibility, Command, Iterator, Mediator, Memento, Observer, State, Strategy, Template Method, Visitor.
          Патерн Composite належить до групи структурних патернів.
     3.2 Призначення патерну Composite
     Composite компонує обʼєкти у деревоподібні структури для представлення ієрархій типу «частина — ціле». Головна ідея полягає в тому, що клієнтський код працює з окремими обʼєктами та їхніми композиціями однаково через єдиний інтерфейс.
     Ключові аспекти:
     • Єдиний інтерфейс для листків та композитів;
     • Рекурсивна деревоподібна структура;
     • Прозорість для клієнтського коду;
     • Спрощення додавання нових типів компонентів.
     3.3 Структура патерну (UML-діаграма)
     Інтерфейс Component оголошує спільні операції — operation, add, remove, getChild. Client працює з обʼєктами через цей інтерфейс. Leaf реалізує інтерфейс і виконує базову дію. Composite також реалізує інтерфейс, але зберігає колекцію дочірніх компонентів children. Він делегує виконання операції кожному зі своїх дочірніх елементів рекурсивно. Звʼязок агрегації показує, що Composite містить нуль або більше Component.
     3.4 Учасники патерну
     Component — оголошує спільний інтерфейс для всіх обʼєктів у композиції, визначає операції для доступу та управління дочірніми компонентами.
     Leaf — представляє листковий обʼєкт без дочірніх елементів, реалізує базову поведінку операції.
     Composite — зберігає дочірні компоненти та реалізує операції, повʼязані з ними, делегує виконання операції дочірнім обʼєктам.
     Client — працює з обʼєктами композиції через інтерфейс Component і не залежить від конкретних класі
     
     3.5 Коли застосовувати Composite
     Використання патерну доцільне коли:
     • Потрібно представити деревоподібну ієрархію обʼєктів «частина — ціле»;
     • Клієнт повинен однаково працювати з простими та складеними обʼєктами;
     • При роботі з рекурсивними структурами: UI-компоненти, файлові системи, меню, організаційні структури;
     • Композиція обʼєктів змінюється динамічно під час виконання програми.
     
     3.6 Реалізація патерну на TypeScript
     Для демонстрації патерну обрано приклад файлової системи. Інтерфейс FileSystemComponent оголошує методи getName, getSize та display. Клас File реалізує інтерфейс як листковий елемент, а клас Directory реалізує Composite, зберігаючи масив дочірніх компонентів.
     Приклад інтерфейсу та Leaf:
1 interface FileSystemComponent {
2 getName(): string;
3 getSize(): number;
4 display(indent: string): void;
5 }
6
7 class File implements FileSystemComponent {
8 constructor(
9 private name: string,
10 private size: number
11 ) {}
12
13 getName(): string { return this.name; }
14 getSize(): number { return this.size; }
15
16 display(indent: string = ""): void {
17 console.log(`${indent} ${this.name} (${this.size} KB)`);
18 }
19 }
     Приклад Composite:
1 class Directory implements FileSystemComponent {
2 private children: FileSystemComponent[] = [];
3
4 constructor(private name: string) {}
5
6 add(component: FileSystemComponent): void {
7 this.children.push(component);
8 }
9
10 remove(component: FileSystemComponent): void {
11 const idx = this.children.indexOf(component);
12 if (idx !== -1) this.children.splice(idx, 1);
13 }
14
15 getName(): string { return this.name; }
16
17 getSize(): number {
18 return this.children.reduce(
19 (sum, child) => sum + child.getSize(), 0
20 );
21 }
22
23 display(indent: string = ""): void {
24 console.log(`${indent} ${this.name} (${this.getSize()} KB)`);
25 this.children.forEach(c => c.display(indent + " "));
26 }
27 }
     
     3.7 Переваги та недоліки
     Переваги:
     • Спрощує клієнтський код завдяки єдиному інтерфейсу;
     • Легко додавати нові типи компонентів без зміни існуючого коду;
     • Підтримує принцип Open/Closed;
     • Забезпечує природну рекурсивну обробку ієрархій.
     Недоліки:
     • Складно обмежити типи дочірніх компонентів на рівні типізації;
     • Загальний інтерфейс може бути надлишковим для Leaf;
     • Ускладнює типізацію у статичних мовах;
     • Можливе зниження продуктивності на глибоких деревах через рекурсію.
     3.8 Реальне застосування
     Composite широко застосовується у сучасній розробці:
     • React / UI-компоненти — дерево компонентів React, де кожен компонент може бути як листком, так і контейнером для дочірніх;
     • Файлова система — файли (Leaf) та директорії (Composite) реалізують спільний інтерфейс для обходу та обчислення розміру;
     • DOM-дерево — HTML-елементи утворюють деревоподібну ієрархію з єдиним API для маніпуляцій;
     • Графічні редактори — фігури та групи фігур оброблюються однаково при переміщенні, масштабуванні, малюванні.


     
     4 ВИСНОВКИ
     У результаті виконання практичного заняття було досліджено патерн Composite, проаналізовано його архітектурні особливості, реалізовано приклади програмного коду на TypeScript. Composite є структурним патерном для роботи з деревоподібними ієрархіями, він забезпечує єдиний інтерфейс для листків та композитів, широко застосовується у UI, файлових системах, DOM та графіці, спрощує клієнтський код і відповідає принципу Open/Closed.
