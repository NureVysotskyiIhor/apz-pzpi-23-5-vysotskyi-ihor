     1. АНАЛІЗ ПРОБЛЕМИ ТА АНАЛОГІВ
     1.1 Аналіз проблеми
     Опис проблеми:
     Сучасні організації, освітні заклади та бізнес-структури щоденно стикаються з необхідністю швидкого збору думок та прийняття колективних рішень. Голосування, опитування студентів, опитування на конференціях, прийняття рішень у командах — всі ці ситуації вимагають ефективного інструменту для швидкої організації та проведення голосувань.
     Чому проблема важлива:
     1. Втрата часу на організацію: Традиційні методи вимагають значних часових витрат. Організатор витрачає до 1-2 годин на організацію одного голосування.
     2. Відсутність оперативності: Затримки в отриманні результатів можуть призвести до втрати можливостей або неправильних рішень.
     3. Низька залученість учасників: Складні процедури знижують мотивацію людей брати участь. Середня відповідь на email-опитування становить 10-30%.
     4. Відсутність real-time результатів: Учасники не можуть миттєво побачити результати, що знижує довіру до процесу.
     5. Проблеми доступності: Традиційні методи вимагають фізичної присутності або складних технічних налаштувань.
     Як поточні підходи не задовольняють потреб:
     • Паперові голосування: вимагають присутності, тривалий підрахунок, без миттєвих результатів
     • Email-опитування: низька активність, без візуалізації результатів
     • Google Forms: складні для швидкого створення, без real-time можливостей
     • Mentimeter, Slido: дорогі ($10-50/міс), орієнтовані на презентації


     1.2 Аналіз аналогів
     Google Forms
     Переваги: безкоштовний, необмежена кількість відповідей
Недоліки: без real-time візуалізації, складний інтерфейс, результати оновлюються лише при перезавантаженні
Що не вирішує: не підходить для швидких голосувань під час зустрічей
     Mentimeter
     Переваги: професійна візуалізація, real-time оновлення
Недоліки: $11.99/міс, безкоштовна версія — тільки 2 питання, орієнтована на презентації
Що не вирішує: не підходить для щоденних потреб через вартість
     Slido
     Переваги: спеціалізація на подіях, real-time результати
Недоліки: від $10/event, безкоштовна версія — 100 учасників, складна настройка
Що не вирішує: не підходить для малих команд та регулярних опитувань
     Як нова система покращить ситуацію:
     • Швидкість: миттєве створення голосування за 30 секунд без складних налаштувань
     • Real-time аналітика: учасники бачать результати в режимі реального часу
     • Простота: мінімалістичний інтерфейс, без обов'язкової реєстрації
     • Доступність: вебверсія, QR-коди для швидкого доступу
     • Доступна ціна: безкоштовна основна функціональність

     2. БАЧЕННЯ РІШЕННЯ
     2.1 Опис призначення системи
     Що робить система:
     Система дозволяє організаторам швидко створювати голосування та опитування без реєстрації, а учасникам — голосувати та бачити результати у режимі реального часу.
     Яку цінність створює для користувачів:
     • Для організаторів: економія часу на організацію, оперативні результати, можливість збереження своїх голосувань
     • Для учасників: зручний швидкий процес голосування, real-time результати, підвищена залученість
     Цільова аудиторія:
     • Викладачі та навчальні заклади
     • Корпоративні організації
     • Організатори подій та конференцій
     • Маркетологи та дослідники
     2.2 Опис бізнес-логіки
     Як працюватиме система:
     1. Створення голосування: Організатор заповнює форму з питанням, типом голосування та варіантами відповідей. Система генерує унікальне посилання та QR-код. Дані організатора зберігаються для подальшого доступу до своїх голосувань.
     2. Доступ до голосування: Учасники використовують посилання або сканують QR-код. Реєстрація не потрібна.
     3. Голосування: Користувач вибирає відповідь та голосує. Система перевіряє, чи він вже голосував (за device fingerprint/IP-адресою).
     4. Результати в реальному часі: Результати оновлюються миттєво для всіх підключених учасників через WebSocket.
     5. Управління голосуванням: Організатор може закрити голосування та експортувати результати.
     6. Адміністративні функції: Система передбачає окремий адміністраторський доступ через JWT-токен (видається окремо адміністратору без реєстрації). Адміністратор має права на: перегляд усіх голосувань системи, примусове закриття/архівування голосувань, блокування пристроїв (device fingerprint), перегляд детальної статистики системи, експорт даних усіх голосувань, переглядання логів дій.
     Які дії виконуватимуть користувачі:
     • Організатор: створює голосування, отримує посилання/QR, закриває голосування, переглядає результати, експортує дані, переглядає історію своїх голосувань
     • Учасник: сканує QR або переходить за посиланням, голосує, бачить результати, не може голосувати двічі
     • Адміністратор: логується за JWT-токеном в захищену панель, переглядає всі голосування системи, може закрити/видалити/архівувати будь-яке голосування, блокує пристрої та device fingerprints, переглядає детальну статистику системи та логи операцій, експортує дані усіх голосувань


     3. ФУНКЦІОНАЛЬНІСТЬ КОМПОНЕНТІВ
     3.1 Scope of Initial Release
(Обсяг першого релізу)

Перший реліз охоплює чотири компоненти: серверну частину,
веб-клієнт, мобільний клієнт та IoT-клієнт. Нижче наведено
детальний опис кожного з них на основі реалізованого коду.


  3.1.1 Серверна частина (polls-backend)

  Технологічний стек: Java 21, Spring Boot 3.5.7, PostgreSQL, Maven. Документація API — Springdoc OpenAPI (Swagger UI).
     REST API охоплює п'ять груп ендпоінтів:
     • /api/polls — повний CRUD голосувань, фільтрація за статусом (ACTIVE / CLOSED / ARCHIVED), сортування за активністю (trending), статистика та метрики розподілу (mean, stdDeviation, coefficientOfVariation), генерація QR-коду, а також експорт результатів у CSV та PDF (iTextPDF 7).
     • /api/votes — реєстрація голосу (pollId, optionId, fingerprintId), перелік та видалення голосів адміністратором.
     • /api/device-fingerprints — реєстрація пристрою, перегляд інформації, блокування та розблокування.
     • /api/admins — CRUD-операції для адміністраторів та статистика активності за період.
     • /api/admin/backups — створення, перегляд, відновлення та видалення резервних копій БД.
     Модель даних: Система оперує шістьма основними сутностями. Poll зберігає заголовок, питання, тип (SINGLE / MULTIPLE / RATING / OPEN), статус та налаштування видимості результатів. PollOption — варіант відповіді з текстом і порядковим номером. Vote — запис голосу з прив'язкою до голосування, варіанта та пристрою, а для OPEN-типу — з текстовою відповіддю. DeviceFingerprint — унікальний хеш пристрою, IP-адреса (inet), user-agent та статус блокування. Admin — ідентифікатор адміністратора. AdminLog — журнал дій адміністратора (CLOSE_POLL, DELETE_VOTE тощо).
     Антифрод-механізм: Метод hasAlreadyVoted() блокує повторне голосування: один fingerprint — один голос у конкретному голосуванні. Метод calculateAnomalyScore() обчислює співвідношення голосів з одного пристрою до середнього показника; при значенні > 3.0 система сигналізує адміністратору.
     Real-time трансляція (WebSocket): Клієнт підключається та підписується на топік polls/{pollId}/results. При кожному новому або видаленому голосі WebSocketBroadcaster розсилає оновлену статистику усім підписникам через ConcurrentHashMap<String, Set<WebSocketSession>>.
     QR-код: QrCodeService генерує посилання на голосування та повертає URL QR-зображення через Google Charts API.
     
  3.1.2 Веб-клієнт
  Технологічний стек: React + Vite, TanStack Router, Zustand, Tailwind CSS, Shadcn/UI, Axios, SockJS + STOMP, lucide-react.
     Реалізує три ролі:
     • Організатор — створення голосувань, отримання посилання та QR-коду, трансляція результатів у реальному часі, детальна статистика, закриття голосування, експорт у CSV та PDF.
     • Учасник — простий інтерфейс голосування (один або кілька варіантів, текстова відповідь для OPEN), real-time діаграма результатів (якщо showResults=true), блокування повторного голосування.
     • Адміністратор — захищена JWT-панель, управління усіма голосуваннями, блокування пристроїв, перегляд логів, резервне копіювання БД, системна статистика.
     Взаємодія з сервером: REST API (Axios) + WebSocket (SockJS + STOMP) для підписки на polls/{id}/results.
     


  3.1.3 Мобільний клієнт

  Спрощена версія веб-клієнта, оптимізована для смартфонів. Ключові відмінності: сканування QR-коду вбудованою камерою, тач-інтерфейс голосування, push-сповіщення через Firebase Cloud Messaging (опціонально), локальний кеш останніх голосувань. Device fingerprint формується на основі ідентифікатора мобільного пристрою. Взаємодія: REST API + WebSocket.

  Взаємодія з сервером:
    • REST API для операцій (той самий backend).
    • WebSocket для real-time результатів.
    • Device fingerprint формується на основі унікального
      ідентифікатора мобільного пристрою.


  3.1.4 IoT-клієнт

  Фізичний пристрій (при фінансуванні) для голосування у конференц-залах та аудиторіях. Апаратна база: ESP32 або Raspberry Pi Zero 2W з WiFi, 2–6 фізичних кнопок та OLED-дисплей (128×64). Пристрій реєструє свій fingerprint, прив'язується до голосування, відображає питання на дисплеї та відправляє голос при натисканні кнопки. Захист від повторного голосування — той самий механізм hasAlreadyVoted() серверу.

3.2 Scope of Subsequent Releases
(Інтеграція штучного інтелекту)

Другий і наступні релізи зосереджені на впровадженні ШІ
для підвищення якості системи. Нижче описано конкретні
завдання, джерела даних та технічні підходи.


  3.2.1 Застосування ШІ для покращення системи

  Заплановано п'ять напрямків інтеграції ШІ:
     • Fraud Detection — заміна поточної евристики (anomalyScore > 3.0) на ML-класифікатор з ознаками: anomalyScore, частота голосів за годину, burst voting, розбіжності IP/User-Agent, збіг з headless-браузерами.
     • NLP для OPEN-відповідей — кластеризація за темами, sentiment analysis, виділення ключових слів та автоматичний підсумок.
     • Рекомендація варіантів — пропонування релевантних варіантів відповідей на основі заголовку питання.
     • Early Prediction — прогнозування фінального розподілу за першими 20–30% голосів.
     • Персоналізовані рекомендації — пропонування релевантних голосувань учаснику за його fingerprint-профілем.
     


  3.2.2 Збір даних для навчання моделей

  Наявні таблиці вже містять необхідні дані: votes (паттерни вибору), device_fingerprints (основа fraud detection; is_blocked — готові мітки), polls (кластеризація тематик), admin_logs (підтверджені факти шахрайства). Додатково рекомендовано збирати: час між послідовними голосами, час перебування на сторінці та результат перевірки is_anomalous.


  3.2.3 Підхід до навчання моделей

  Фаза 1 (API): Claude/GPT-4o для аналізу тексту, LLM з few-shot промптом для рекомендацій варіантів. Швидкий старт без датасету. Фаза 2 (Fine-tuning): XGBoost/LightGBM для fraud detection на accumulated даних за 3–6 місяців; distilBERT для NLP при >10 000 прикладів. Фаза 3 (Власна модель): end-to-end fraud detection з ембедингами, collaborative filtering для рекомендацій. Пріоритет: Фаза 1 (текст) + Фаза 2 (fraud) — найшвидший результат.

3.3 Limitations and Exclusions
(Обмеження та виключення)

Нижче перераховано неочевидні обмеження, що є свідомими
архітектурними рішеннями, та функціонал, явно виключений
зі скоупу.


  3.3.1 Обмеження, пов'язані з ідентифікацією

  Device fingerprint ідентифікує пристрій, а не особу — різні браузери, інкогніто чи VPN дозволяють повторне голосування. Один fingerprint — один голос; зміна вибору не підтримується. IP зберігається як допоміжна ознака, а не основний ключ.


  3.3.2 Обмеження реального часу

WebSocket без STOMP-брокера: немає горизонтального масштабування між інстансами, немає черги пропущених повідомлень. Офлайн-режим не підтримується.


  3.3.3 Обмеження функціоналу голосування

  OPEN-тип — лише зберігання без обробки. Немає планування голосувань, мультимедіа у варіантах та часткової анонімної статистики при showResults=false.


  3.3.4 Явно виключений функціонал

  Свідомо виключено: реєстрація учасників (замість неї — fingerprint), платіжна система, інтеграція з Slack/Teams/Zoom/LMS, інтернаціоналізація (i18n), геолокаційна аналітика, email/SMS/push-сповіщення.


     4. ТЕХНОЛОГІЧНИЙ СТЕК
     4.3 Опис програмних засобів та інструментів
     Розробка програмної системи здійснюватиметься із використанням сучасних технологій серверної, клієнтської та мобільної розробки. Основою серверної частини є Java 21 у поєднанні з Spring Boot 3.5.7, що забезпечує надійність, масштабованість та чітку структуру проєкту.
     Серверна логіка реалізовуватиметься на базі Spring Web для створення REST-інтерфейсів, а робота з базою даних виконуватиметься за допомогою Spring Data JPA, що надає зручний ORM-шар та спрощує взаємодію з PostgreSQL. Для забезпечення обміну даними в реальному часі використовуватиметься Spring WebSocket зі структурою STOMP. Документація API формуватиметься за допомогою Springdoc OpenAPI, що надає інтерфейс Swagger UI. Управління залежностями та збірка проєкту виконуватимуться за допомогою Maven. Spring Security буде застосовано для аутентифікації адміністратора через JWT-токени та управління ролями доступу (ROLE_ADMIN). Система реалізуватиме рольову модель контролю доступу (RBAC) для розрізнення прав організаторів, учасників та адміністраторів.
     Клієнтська частина (веб) реалізовуватиметься засобами React, що дозволить створити швидкий та адаптивний інтерфейс. Для управління станом застосовуватиметься Zustand, а стилізація інтерфейсу виконуватиметься за допомогою Tailwind CSS та компонентів Shadcn/UI. Маршрутизація забезпечуватиметься TanStack Router, а для real-time оновлень застосовуватимуться SockJS та STOMP. Взаємодія з HTTP-інтерфейсами відбуватиметься через Axios, а інструментом розробки та збірки слугуватиме Vite. Для відображення векторних іконок застосовуватиметься бібліотека lucide-react.
     Мобільний клієнт для iOS реалізовуватиметься мовою Swift з використанням фреймворку SwiftUI для побудови декларативного інтерфейсу користувача. Архітектурним підходом обрано MVVM (Model-View-ViewModel), що забезпечує чітке розділення бізнес-логіки та представлення. Мережева взаємодія з серверним REST API здійснюватиметься через URLSession. Для підтримки оновлень у реальному часі використовуватиметься URLSessionWebSocketTask, що забезпечує нативне WebSocket-з'єднання без сторонніх залежностей. Сканування QR-кодів реалізовуватиметься за допомогою фреймворку AVFoundation, що надає доступ до камери пристрою. Мінімальна підтримувана версія — iOS 16. Для локального кешування даних застосовуватиметься Core Data, а ідентифікація пристрою забезпечуватиметься через UIDevice.identifierForVendor як основу для формування device fingerprint.
     Проєкт розроблятиметься із використанням системи контролю версій Git, а код зберігатиметься у GitHub. Основними інструментами в робочому процесі будуть Visual Studio Code для веб-клієнтської частини, IntelliJ IDEA для серверної та Xcode для мобільного iOS-клієнта. Для тестування API застосовуватиметься Postman, а контейнеризація серверної частини забезпечуватиметься Docker.
     Система повинна гарантувати стабільну роботу та високу швидкодію під час обробки запитів кількох користувачів. Доступ до неї здійснюватиметься через вебінтерфейс та мобільний додаток, а передача даних відбуватиметься через захищений канал HTTPS. Надійність забезпечуватиметься використанням перевірених технологій, резервним копіюванням даних та чіткою організацією взаємодії компонентів. Продуктивність досягатиметься завдяки ефективній роботі Spring Boot та PostgreSQL, а цілісність даних гарантуватиметься через механізми транзакцій та обмежень у СУБД.
