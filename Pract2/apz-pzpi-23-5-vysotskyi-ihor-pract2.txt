     3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Загальні відомості про систему
     Discord — комунікаційна платформа для текстового, голосового та відеозв'язку. Розроблена у 2015 році компанією Discord Inc. (Сан-Франциско, США). Спочатку орієнтована на ігрові спільноти, зараз використовується для освіти, бізнесу та відкритих проєктів. Платформа обслуговує понад 150 мільйонів активних користувачів на місяць, 19 мільйонів активних серверів щодня та понад 4 мільярди хвилин голосового зв'язку на день. Клієнтські додатки: Desktop (Electron), Mobile (React Native), Web (React).
     3.2 Технологічний стек
     Discord використовує полігот-підхід (polyglot) — різні мови програмування для різних задач:
     • Real-time Gateway: Elixir / Erlang (BEAM VM) — масивна конкурентність, мільйони WebSocket з'єднань;
     • API Monolith: Python (Django) — швидка ітерація, бізнес-логіка, REST API;
     • Голосовий стек: Rust + WebRTC + C++ — низька затримка, обробка аудіо без GC;
     • Зберігання повідомлень: ScyllaDB (ex-Cassandra) — трильйони повідомлень, горизонтальне масштабування;
     • Кешування / Presence: Redis — статуси користувачів, сесії, pub/sub;
     • Метадані / Auth: PostgreSQL — реляційні дані: акаунти, сервери, ролі;
     • Інфраструктура: GCP + власні сервери — гібридна хмарна модель.
     3.3 Архітектурний стиль
     Discord використовує гібридну мікросервісну архітектуру. Python API побудований як моноліт для зручності розробки. Elixir-стек містить 20+ мікросервісів для real-time обробки, кожен масштабується незалежно. Rust застосовується для спеціалізованих сервісів — голосовий сервер та NIF-модулі для Elixir. Сервіси обмінюються подіями через pub/sub (Redis, внутрішні черги) в асинхронному режимі.
     3.4 Компоненти системи та їх взаємодія
     Клієнт підключається до API Gateway через WebSocket. Gateway маршрутизує запити до відповідного мікросервісу: Guild Service (Elixir), Message Service (Python), Voice Service (Rust), Presence Service (Elixir). Кожен мікросервіс звертається до відповідного сховища даних: ScyllaDB для повідомлень, PostgreSQL для метаданих, Redis для кешування та pub/sub. Голосові дзвінки обробляються через окремий Voice Gateway, що направляє аудіо на виділений Voice Server (Rust, UDP/WebRTC).
     3.5 Real-time: Elixir та BEAM VM
     BEAM VM забезпечує мільйони легковагих процесів. Кожен guild (сервер) — окремий GenServer-процес. Система обробляє 26 мільйонів WebSocket подій на секунду та підтримує 12 мільйонів одночасних користувачів. BEAM надає вбудовану відмовостійкість через supervisor tree та гарячу заміну коду без зупинки.
     Приклад обробки повідомлень у Guild (Elixir):
1 defmodule Discord.Guild do
2 use GenServer
3
4 def handle_cast({:message, msg}, state) do
5 Enum.each(state.members, fn pid ->
6 send(pid, {:new_message, msg})
7 end)
8 {:noreply, state}
9 end
10 end
     3.6 Голосовий стек: Rust та WebRTC
     Голосовий стек побудований на Rust для забезпечення передбачуваної затримки без GC-пауз. Обробка аудіо використовує кодек Opus та мікшування. WebRTC через UDP забезпечує мінімальну затримку з SFU-архітектурою (Selective Forwarding Unit). Критичні операції в Elixir (SortedSet для списків учасників) працюють через Rust NIF — в 160 разів швидше за чистий Elixir.
     3.7 Зберігання даних: від Cassandra до ScyllaDB
     Еволюція сховища повідомлень:
     • 2015: MongoDB — швидкий старт, але не масштабується;
     • 2017: Apache Cassandra — 12 нод, мільярди повідомлень;
     • 2022: Cassandra — 177 нод, трильйони повідомлень, GC-паузи, latency spikes;
     • 2023: ScyllaDB — C++ замість Java, shard-per-core, P99 latency 185мс → 1.4мс, вдвічі менше нод.
     Схема зберігання повідомлень (CQL):
1 CREATE TABLE messages (
2 channel_id bigint, bucket int,
3 message_id bigint, author_id bigint,
4 content text,
5 PRIMARY KEY ((channel_id, bucket), message_id)
6 ) WITH CLUSTERING ORDER BY (message_id DESC);
     3.8 Проблеми: приватність та безпека
     Незважаючи на технічні досягнення, Discord має серйозні проблеми:
     • Відсутність E2E-шифрування — усі повідомлення проходять через сервери у відкритому вигляді;
     • Збір даних без згоди — трекінг через /api/track та /api/science навіть без дозволу користувача;
     • Витоки даних (2023, 2025) — зламано сторонній сервіс підтримки, витекли паспортні дані;
     • «Чорна діра знань» — контент серверів не індексується пошуковими системами;
     • Зміни Privacy Policy для AI — дані передаються OpenAI для AI-функцій без належних гарантій.
     3.9 Переваги та недоліки архітектури
     Сильні сторони: Elixir + BEAM — ідеальна конкурентність; Rust NIF — 160x приріст продуктивності; ScyllaDB — P99 latency 1.4мс; масштабування до 15M+ юзерів на одному сервері; полігот-стек — правильний інструмент для задачі.
     Слабкі сторони: відсутність E2E-шифрування повідомлень; агресивний збір даних користувачів; закритий код — неможливо аудит; Python-моноліт як bottleneck; залежність від пропрієтарної платформи.
     



     
     4 ВИСНОВКИ
     У результаті виконання практичного заняття було досліджено архітектуру Discord — полігот-мікросервісної системи на Elixir, Rust та Python. Інженерні рішення (BEAM, Rust NIF, ScyllaDB) дозволили масштабуватися до 150M+ користувачів. Водночас виявлено серйозні проблеми з приватністю та безпекою. Архітектура Discord є цінним кейсом для вивчення як технічних переваг, так і етичних компромісів.
