Інформацію з інженерного блогу Discord та сторонніх технічних джерел було верифіковано та зіставлено між собою для забезпечення достовірності. Приклади коду взаємодії з Discord API перевірено на відповідність офіційній документації Discord Developer Portal (v10). Концептуальні приклади внутрішньої архітектури (Elixir, Rust, CQL) перевірено на відповідність описам з офіційних публікацій Discord Engineering Blog. Діаграму архітектури системи було перевірено та скориговано щодо коректності потоків даних між компонентами. Текст доповіді вичитано та адаптовано під науково-технічний стиль відповідно до вимог ДСТУ 3008:2015. Відеозапис доповіді записано та озвучено автором самостійно.
     
     3.1 Загальні відомості про систему
     Discord — комунікаційна платформа для текстового, голосового та відеозв'язку. Розроблена у 2015 році компанією Discord Inc. (Сан-Франциско, США). Спочатку орієнтована на ігрові спільноти, зараз використовується для освіти, бізнесу та відкритих проєктів. Платформа обслуговує понад 150 мільйонів активних користувачів на місяць, 19 мільйонів активних серверів щодня та понад 4 мільярди хвилин голосового зв'язку на день. Клієнтські додатки: Desktop (Electron), Mobile (React Native), Web (React).
     3.2 Технологічний стек
     Discord використовує полігот-підхід (polyglot) — різні мови програмування для різних задач:
     • Real-time Gateway: Elixir / Erlang (BEAM VM) — масивна конкурентність, мільйони WebSocket з'єднань;
     • API Monolith: Python (Django) — швидка ітерація, бізнес-логіка, REST API;
     • Голосовий стек: Rust + WebRTC + C++ — низька затримка, обробка аудіо без GC;
     • Зберігання повідомлень: ScyllaDB (ex-Cassandra) — трильйони повідомлень, горизонтальне масштабування;
     • Кешування / Presence: Redis — статуси користувачів, сесії, pub/sub;
     • Метадані / Auth: PostgreSQL — реляційні дані: акаунти, сервери, ролі;
     • Інфраструктура: GCP + власні сервери — гібридна хмарна модель.
     3.3 Архітектурний стиль
     Discord використовує гібридну мікросервісну архітектуру. Python API побудований як моноліт для зручності розробки. Elixir-стек містить 20+ мікросервісів для real-time обробки, кожен масштабується незалежно. Rust застосовується для спеціалізованих сервісів — голосовий сервер та NIF-модулі для Elixir. Сервіси обмінюються подіями через pub/sub (Redis, внутрішні черги) в асинхронному режимі.
     3.4 Компоненти системи та їх взаємодія
     Клієнт підключається до API Gateway через WebSocket. Gateway маршрутизує запити до відповідного мікросервісу: Guild Service (Elixir), Message Service (Python), Voice Service (Rust), Presence Service (Elixir). Кожен мікросервіс звертається до відповідного сховища даних: ScyllaDB для повідомлень, PostgreSQL для метаданих, Redis для кешування та pub/sub. Голосові дзвінки обробляються через окремий Voice Gateway, що направляє аудіо на виділений Voice Server (Rust, UDP/WebRTC).
     3.5 Real-time: Elixir та BEAM VM
     BEAM VM забезпечує мільйони легковагих процесів. Кожен guild (сервер) — окремий GenServer-процес. Система обробляє 26 мільйонів WebSocket подій на секунду та підтримує 12 мільйонів одночасних користувачів. BEAM надає вбудовану відмовостійкість через supervisor tree та гарячу заміну коду без зупинки.
     Приклад обробки повідомлень у Guild (Elixir):
1 defmodule Discord.Guild do
2 use GenServer
3
4 def handle_cast({:message, msg}, state) do
5 Enum.each(state.members, fn pid ->
6 send(pid, {:new_message, msg})
7 end)
8 {:noreply, state}
9 end
10 end
     3.6 Голосовий стек: Rust та WebRTC
     Голосовий стек побудований на Rust для забезпечення передбачуваної затримки без GC-пауз. Обробка аудіо використовує кодек Opus та мікшування. WebRTC через UDP забезпечує мінімальну затримку з SFU-архітектурою (Selective Forwarding Unit). Критичні операції в Elixir (SortedSet для списків учасників) працюють через Rust NIF — в 160 разів швидше за чистий Elixir.
     3.7 Зберігання даних: від Cassandra до ScyllaDB
     Еволюція сховища повідомлень:
     • 2015: MongoDB — швидкий старт, але не масштабується;
     • 2017: Apache Cassandra — 12 нод, мільярди повідомлень;
     • 2022: Cassandra — 177 нод, трильйони повідомлень, GC-паузи, latency spikes;
     • 2023: ScyllaDB — C++ замість Java, shard-per-core, P99 latency 185мс → 1.4мс, вдвічі менше нод.
     Схема зберігання повідомлень (CQL):
1 CREATE TABLE messages (
2 channel_id bigint, bucket int,
3 message_id bigint, author_id bigint,
4 content text,
5 PRIMARY KEY ((channel_id, bucket), message_id)
6 ) WITH CLUSTERING ORDER BY (message_id DESC);
     3.8 Проблеми: приватність та безпека
     Незважаючи на технічні досягнення, Discord має серйозні проблеми:
     • Відсутність E2E-шифрування — усі повідомлення проходять через сервери у відкритому вигляді;
     • Збір даних без згоди — трекінг через /api/track та /api/science навіть без дозволу користувача;
     • Витоки даних (2023, 2025) — зламано сторонній сервіс підтримки, витекли паспортні дані;
     • «Чорна діра знань» — контент серверів не індексується пошуковими системами;
     • Зміни Privacy Policy для AI — дані передаються OpenAI для AI-функцій без належних гарантій.
     3.9 Переваги та недоліки архітектури
     Сильні сторони: Elixir + BEAM — ідеальна конкурентність; Rust NIF — 160x приріст продуктивності; ScyllaDB — P99 latency 1.4мс; масштабування до 15M+ юзерів на одному сервері; полігот-стек — правильний інструмент для задачі.
     Слабкі сторони: відсутність E2E-шифрування повідомлень; агресивний збір даних користувачів; закритий код — неможливо аудит; Python-моноліт як bottleneck; залежність від пропрієтарної платформи.

Інженерні рішення (BEAM, Rust NIF, ScyllaDB) дозволили масштабуватися до 150M+ користувачів. Водночас виявлено серйозні проблеми з приватністю та безпекою. Архітектура Discord є цінним кейсом для вивчення як технічних переваг, так і етичних компромісів.

2. // Архітектура Discord: Приклади програмного коду
3. // Демонстрація архітектурних принципів через публічне API
4. // Мови: TypeScript, Elixir, CQL (Cassandra/ScyllaDB)

6. // ============================================================
7. // 1. Gateway WebSocket — підключення до Discord Gateway
8. //    Демонструє: WebSocket-архітектуру, heartbeat-механізм,
9. //    event-driven підхід, opcode-протокол
10. // ============================================================

11. import WebSocket from "ws";

12. const DISCORD_GATEWAY_URL = "wss://gateway.discord.gg/?v=10&encoding=json";

13. // Opcodes Discord Gateway протоколу
14. enum GatewayOpcode {
15. Dispatch = 0,         // Подія від сервера
16. Heartbeat = 1,        // Heartbeat (клієнт → сервер)
17. Identify = 2,         // Ідентифікація при підключенні
18. Resume = 6,           // Відновлення з'єднання
19. Reconnect = 7,        // Сервер вимагає перепідключення
20. InvalidSession = 9,   // Невалідна сесія
21. Hello = 10,           // Перше повідомлення від сервера
22. HeartbeatAck = 11,    // Підтвердження heartbeat
23. }

24. interface GatewayPayload {
25. op: GatewayOpcode;
26. d: unknown;
27. s: number | null;     // Sequence number для resume
28. t: string | null;     // Event name (тільки для op=0)
29. }

30. class DiscordGatewayClient {
31. private ws: WebSocket | null = null;
32. private heartbeatInterval: NodeJS.Timeout | null = null;
33. private lastSequence: number | null = null;
34. private token: string;

35. constructor(token: string) {
36. this.token = token;
37. }

38. connect(): void {
39. this.ws = new WebSocket(DISCORD_GATEWAY_URL);

40. this.ws.on("open", () => {
41. console.log("[Gateway] З'єднання встановлено");
42. });

43. this.ws.on("message", (data: string) => {
44. const payload: GatewayPayload = JSON.parse(data);
45. this.handlePayload(payload);
46. });

47. this.ws.on("close", (code: number) => {
48. console.log(`[Gateway] З'єднання закрито: ${code}`);
49. this.stopHeartbeat();
50. });
51. }

52. private handlePayload(payload: GatewayPayload): void {
53. // Зберігаємо sequence для можливого resume
54. if (payload.s !== null) {
55. this.lastSequence = payload.s;
56. }

57. switch (payload.op) {
58. case GatewayOpcode.Hello:
59. // Перше повідомлення — починаємо heartbeat
60. const { heartbeat_interval } = payload.d as {
61. heartbeat_interval: number;
62. };
63. this.startHeartbeat(heartbeat_interval);
64. this.identify();
65. break;

66. case GatewayOpcode.HeartbeatAck:
67. console.log("[Gateway] Heartbeat ACK отримано");
68. break;

69. case GatewayOpcode.Dispatch:
70. // Event-driven: обробка подій від Discord
71. this.handleEvent(payload.t!, payload.d);
72. break;

73. case GatewayOpcode.Reconnect:
74. console.log("[Gateway] Сервер вимагає перепідключення");
75. this.reconnect();
76. break;
77. }
78. }

79. // Heartbeat — механізм підтримки з'єднання
80. // Discord закриє WebSocket якщо не отримає heartbeat
81. private startHeartbeat(intervalMs: number): void {
82. this.sendHeartbeat();
83. this.heartbeatInterval = setInterval(() => {
84. this.sendHeartbeat();
85. }, intervalMs);
86. }

87. private sendHeartbeat(): void {
88. this.send({ op: GatewayOpcode.Heartbeat, d: this.lastSequence });
89. }

90. private stopHeartbeat(): void {
91. if (this.heartbeatInterval) {
92. clearInterval(this.heartbeatInterval);
93. }
94. }

95. // Identify — ідентифікація бота з токеном та intents
96. // Intents визначають які події бот отримуватиме
97. private identify(): void {
98. this.send({
99. op: GatewayOpcode.Identify,
100. d: {
101. token: this.token,
102. intents: 513, // GUILDS | GUILD_MESSAGES
103. properties: {
104. os: "linux",
105. browser: "custom-client",
106. device: "custom-client",
107. },
108. },
109. });
110. }

111. // Event-driven обробка — кожна подія має свій тип
112. private handleEvent(eventName: string, data: unknown): void {
113. switch (eventName) {
114. case "READY":
115. console.log("[Gateway] Бот підключений та готовий");
116. break;
117. case "MESSAGE_CREATE":
118. console.log("[Gateway] Нове повідомлення отримано");
119. break;
120. case "GUILD_CREATE":
121. console.log("[Gateway] Інформація про сервер отримана");
122. break;
123. }
124. }

125. private reconnect(): void {
126. this.ws?.close();
127. this.connect();
128. }

129. private send(data: object): void {
130. this.ws?.send(JSON.stringify(data));
131. }
132. }


133. // ============================================================
134. // 2. REST API — взаємодія з Discord HTTP API
135. //    Демонструє: REST-архітектуру, rate limiting,
136. //    CRUD-операції на ресурсах Discord
137. // ============================================================

138. const DISCORD_API_BASE = "https://discord.com/api/v10";

139. interface DiscordMessage {
140. id: string;
141. channel_id: string;
142. author: { id: string; username: string };
143. content: string;
144. timestamp: string;
145. }

146. interface DiscordChannel {
147. id: string;
148. name: string;
149. type: number;
150. guild_id: string;
151. }

152. class DiscordRestClient {
153. private token: string;

154. constructor(token: string) {
155. this.token = token;
156. }

157. // Базовий HTTP-запит з обробкою Rate Limiting
158. // Discord обмежує кількість запитів (429 Too Many Requests)
159. private async request<T>(
160. method: string,
161. endpoint: string,
162. body?: object
163. ): Promise<T> {
164. const response = await fetch(`${DISCORD_API_BASE}${endpoint}`, {
165. method,
166. headers: {
167. Authorization: `Bot ${this.token}`,
168. "Content-Type": "application/json",
169. },
170. body: body ? JSON.stringify(body) : undefined,
171. });

172. // Rate Limit — архітектурне рішення Discord
173. // для захисту від перевантаження
174. if (response.status === 429) {
175. const retryAfter = response.headers.get("Retry-After");
176. console.log(`[Rate Limit] Очікування ${retryAfter}с`);
177. await new Promise((r) =>
178. setTimeout(r, Number(retryAfter) * 1000)
179. );
180. return this.request<T>(method, endpoint, body);
181. }

182. return response.json() as Promise<T>;
183. }

184. // Отримання повідомлень каналу
185. async getMessages(
186. channelId: string,
187. limit: number = 50
188. ): Promise<DiscordMessage[]> {
189. return this.request<DiscordMessage[]>(
190. "GET",
191. `/channels/${channelId}/messages?limit=${limit}`
192. );
193. }

194. // Надсилання повідомлення
195. async sendMessage(
196. channelId: string,
197. content: string
198. ): Promise<DiscordMessage> {
199. return this.request<DiscordMessage>(
200. "POST",
201. `/channels/${channelId}/messages`,
202. { content }
203. );
204. }

205. // Отримання каналів серверу (guild)
206. async getGuildChannels(
207. guildId: string
208. ): Promise<DiscordChannel[]> {
209. return this.request<DiscordChannel[]>(
210. "GET",
211. `/guilds/${guildId}/channels`
212. );
213. }
214. }


215. // ============================================================
216. // 3. Discord Bot з обробкою подій
217. //    Демонструє: event-driven архітектуру, інтеграцію
218. //    Gateway (WebSocket) + REST API
219. // ============================================================

220. class SimpleDiscordBot {
221. private gateway: DiscordGatewayClient;
222. private rest: DiscordRestClient;

223. constructor(token: string) {
224. this.gateway = new DiscordGatewayClient(token);
225. this.rest = new DiscordRestClient(token);
226. }

227. // Точка входу — підключення до Gateway
228. start(): void {
229. console.log("[Bot] Запуск...");
230. this.gateway.connect();
231. }

341. // 6. Rust NIF: SortedSet для списків учасників
342. //    Демонструє: інтеграцію Rust з Elixir через NIF
343. //    для критичних за продуктивністю операцій
344. // ============================================================

345. /*
346. // Rust: Структура SortedSet для зберігання учасників
347. // Використовується у Discord для member list кожного guild
348. // Забезпечує O(log n) вставку та видалення

349. use rustler::{Encoder, Env, NifResult, Term};

350. struct SortedSet {
351. buckets: Vec<Vec<Term>>,
352. bucket_size: usize,
353. }

354. impl SortedSet {
355. fn new(bucket_size: usize) -> Self {
356. SortedSet {
357. buckets: Vec::new(),
358. bucket_size,
359. }
360. }

361. // Додавання елементу — знаходить потрібний bucket,
362. // виконує binary search всередині
363. fn add(&mut self, term: Term) -> bool {
364. let bucket_idx = self.find_bucket(&term);
365. let bucket = &mut self.buckets[bucket_idx];

366. match bucket.binary_search(&term) {
367. Ok(_) => false,  // Елемент вже існує
368. Err(pos) => {
369. bucket.insert(pos, term);
370. // Розділяємо bucket якщо він переповнений
371. if bucket.len() > self.bucket_size * 2 {
372. self.split_bucket(bucket_idx);
373. }
374. true
375. }
376. }
377. }

378. // Видалення елементу
379. fn remove(&mut self, term: &Term) -> bool {
380. let bucket_idx = self.find_bucket(term);
381. let bucket = &mut self.buckets[bucket_idx];

382. match bucket.binary_search(term) {
383. Ok(pos) => {
384. bucket.remove(pos);
385. true
386. }
387. Err(_) => false,
388. }
389. }
390. }

391. // NIF-функції, доступні з Elixir
392. #[rustler::nif]
393. fn sorted_set_new(bucket_size: usize) -> SortedSet {
394. SortedSet::new(bucket_size)
395. }

396. #[rustler::nif]
397. fn sorted_set_add(set: &mut SortedSet, term: Term)
⇒ bool {
398. set.add(term)
399. }
400. */
